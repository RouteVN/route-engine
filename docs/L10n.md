# L10n Plan

Current RouteEngine runtime does not implement localization.

The old package-based `l10n.keys.*` system has been removed. This document
describes the intended replacement model only. It is not implemented yet.

## Direction

Localization will be patch-based.

- One authored language is the canonical source language.
- The source language lives directly in normal project data.
- Localization packages contain only overrides.
- If an override is missing, the engine shows the original authored value.

This keeps the authoring experience simple:

- authors write the game once in the source language
- translators only provide sparse patches
- missing translations fall back automatically
- there is no duplicated parallel content tree for each language

## Patch Targets

Patches should target stable semantic IDs, not array indexes.

Good targets include:

- `characterId` for character display names
- `lineId` for dialogue or narration content
- `layoutId` + `elementId` for layout text content
- resource IDs for any other authored user-facing text

Avoid:

- array-position based overrides
- generic arbitrary JSON-path patching
- detached translation key bags like `l10n.keys.*`

## Resolution Rule

At render time:

1. Read the original authored value.
2. Check whether the active localization package provides a patch for that exact target.
3. Use the patch if present.
4. Otherwise use the original authored value.

## Template Timing

Localization patches should be applied before template interpolation.

That keeps translated strings compatible with existing templating such as
`${variables.*}`.

## Summary

The intended l10n model is:

- canonical source content in project data
- sparse localization patches keyed by stable IDs
- automatic fallback to original authored content

Implementation will come later.
