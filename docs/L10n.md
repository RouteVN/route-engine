# Localization Feature Plan

Status: proposed replacement for the removed `l10n.keys.*` system. The current
runtime does not implement localization.

This review keeps the previous plan's strongest decision—canonical source data
plus sparse patches—but completes the missing runtime, schema, asset, style,
persistence, fallback, and testing contracts. It follows
[FeatureSystemsArchitecture.md](./FeatureSystemsArchitecture.md).

All localization packages and runtime interfaces are JSON-serializable. There
is no arbitrary JSON Patch, author code, or detached translation-key bag.

## Product Outcome

A project has one canonical source locale authored directly in normal project
data. Each additional locale is a sparse JSON package that replaces only
approved localizable leaves addressed by stable semantic IDs. Missing leaves
fall through an explicit locale fallback chain and ultimately to source data.

Players can switch locale at runtime. Dialogue, choice text, UI text, item and
extras metadata, images/files, fonts, and locale-specific text styles update
without changing story pointers, variables, inventory contents, unlocks, save
slots, or rollback identity.

## Review of the Previous Plan

The previous 65-line plan correctly established:

- one canonical authored source language
- sparse override packages
- stable semantic IDs instead of array indexes
- source fallback for missing overrides
- localization before template interpolation
- rejection of the old `l10n.keys.*` bag

Those decisions remain.

The missing decisions this document adds are:

- locale metadata and fallback rules
- exact patch target families
- stable addressing for lines, choices, nested layout/control elements, and new
  feature catalogs
- translated image/video/spritesheet/audio/font/style resources
- active-locale lifetime and hydration
- runtime locale-switch behavior
- stale/orphan/missing translation tooling
- save/load/rollback interaction
- Route Graphics responsibilities, including RTL limitations
- implementation phases and required test paths

## Research Applied

Ren'Py treats a project as authored in one primary language and supports four
localizable families: dialogue, interface strings, images/files, and styles.
It can replace locale-specific files and rebuild locale-specific font/style
configuration when language changes. Its dialogue translation units may be
split/reordered, which is powerful but depends on an executable script model.
See [Ren'Py Translation](https://www.renpy.org/doc/html/translation.html).

Naninovel similarly has a source locale, locale-specific resources, localized
scripts and managed UI text, runtime selection, locale-specific fonts, and
generated documents with stable IDs. See
[Naninovel Localization](https://naninovel.com/guide/localization) and
[Managed Text](https://naninovel.com/guide/managed-text).

Route Engine is JSON/state driven, so its equivalent should be typed semantic
patch maps. It should not copy executable translation blocks or content-hash IDs
when the project already has stable section, line, element, item, track, and
replay IDs.

## Core Decisions

1. Source content remains in `projectData` exactly where it is authored.
2. Locale packages are sparse and typed by domain.
3. Patch targets use stable IDs and never array positions.
4. A patch may replace only localizable leaves; it cannot alter game logic.
5. Resolution occurs before Jempl/template interpolation.
6. Runtime and saves retain canonical IDs and raw variables, not translated
   render strings.
7. Active locale is a device preference outside save slots and rollback.
8. Route Graphics has no locale registry. It receives resolved text, file
   aliases, and text styles.

## Project Data Contract

Add a top-level `localization` object to project data.

```yaml
localization:
  formatVersion: 1
  sourceLocale: en-US
  defaultLocale: en-US
  locales:
    en-US:
      name: English
      direction: ltr
    ja-JP:
      name: 日本語
      direction: ltr
      packageId: locale-ja-JP
  packages:
    locale-ja-JP:
      formatVersion: 1
      locale: ja-JP
      sourceLocale: en-US
      sourceRevision: sha256-project-source-revision
      patches: {}
      resourceOverrides: {}
      textStyleOverrides: {}
```

### Locale IDs

- Locale IDs use BCP 47-style tags such as `en`, `en-US`, `pt-BR`, or `ja-JP`.
- IDs are case-normalized by tooling and stored in one canonical form.
- `sourceLocale` must exist in `locales` and must not have `packageId`.
- `defaultLocale` must exist and defaults to `sourceLocale`.
- Non-source locales require a package in v1.
- `fallbackLocaleIds` is the complete ordered list of intermediate locales to
  try before source. It must omit the owning locale and `sourceLocale`; the
  runtime appends source exactly once. An omitted or empty list falls directly
  back to source. Referenced locales' own lists are not recursively expanded.
- `name` is the language autonym shown in the selector. It is not translated
  through another locale.
- `direction` is `ltr` or `rtl` and defaults to `ltr`.

### Package Loading

V1 normalizes every declared package into
`projectData.localization.packages` before `engine.init`. Package files may be
stored separately by the packager, but the engine receives JSON objects, not
paths or callbacks.

This keeps the core runtime synchronous and deterministic. Deferred/on-demand
locale packages may be added later with a host loading effect, but must produce
the same normalized package shape before activation.

## Patch Leaf Envelope

Every translated text leaf uses an envelope:

```yaml
value: Translated text
sourceHash: sha256-normalized-source-leaf
```

Array/rich-text leaves preserve their JSON type:

```yaml
value:
  - text: Translated segment
sourceHash: sha256-normalized-source-leaf
```

Rules:

- `value` must have the type required by the target.
- `sourceHash` is generated from canonical JSON serialization of the source
  leaf and is used by tooling to detect stale translations.
- Runtime resolution does not silently discard a translation solely because
  its hash is stale. Build policy reports it as warning or error.
- A null `value` is not a deletion. Omitting the leaf means fallback.
- Empty string is a valid explicit translation for text leaves that permit it.

Stable semantic IDs keep translations attached when source wording changes;
the hash tells translators that the attached value needs review.

## Typed Patch Shape

The package `patches` object is a whitelist of domain maps.

```yaml
patches:
  characters: {}
  story: {}
  layouts: {}
  controls: {}
  items: {}
  inventories: {}
  musicBoxes: {}
  sceneReplays: {}
```

No generic JSON pointer or arbitrary object merge is accepted.

### Characters

```yaml
characters:
  eileen:
    name:
      value: アイリーン
      sourceHash: sha256-...
```

Only display `name` is localizable. `nameVariableId` is logic and cannot be
patched. A runtime name variable remains player data and is not auto-translated.

### Story Lines

Section IDs are globally unique. Line IDs are addressed inside section IDs;
line ID alone is not assumed globally unique.

```yaml
story:
  scenes:
    chapter1:
      name:
        value: 第一章
        sourceHash: sha256-...
  sections:
    chapter1Meeting:
      name:
        value: 出会い
        sourceHash: sha256-...
      lines:
        line30:
          dialogue:
            content:
              value:
                - text: ここで会うとは思わなかった。
              sourceHash: sha256-...
            characterName:
              value: 旅人
              sourceHash: sha256-...
          choices:
            askName:
              content:
                value: 名前を聞く
                sourceHash: sha256-...
          formFields:
            playerName:
              placeholder:
                value: 名前
                sourceHash: sha256-...
          visualText:
            locationLabel:
              content:
                value: 駅前
                sourceHash: sha256-...
```

Rules:

- Dialogue patches replace the complete `dialogue.content` array for the line.
  Current dialogue content items have no nested semantic IDs, so patching array
  positions would be unstable.
- Dialogue `character.name`/compatibility `characterName` may be patched as a
  line-local speaker override.
- Choice items use their authored item `id`. When the top-level `localization`
  object is present, every choice item must have a non-empty `id` that is unique
  among the choice items on that line. Projects without `localization` retain
  the current optional-ID behavior. Enabling localization is a one-time source
  migration: extraction fails with the section/line and offending item until
  authors persist stable IDs in canonical story data; runtime and extraction
  never synthesize IDs from array positions.
- Form placeholders use the authored field ID.
- Visual text items use their authored visual item `id`.
- Scene/section `name` may be patched where it is player-facing.
- Actions, events, routing targets, conditions, variables, resource IDs, and
  timing fields are not localizable.

V1 does not allow translation packages to split, combine, reorder, or omit
story lines. That Ren'Py capability would change Route Engine's pointer,
rollback, voice, and seen-line semantics. If later required, it needs a
separate localized-story-structure design rather than abusing text patches.

### Layout Elements

When the top-level `localization` object is present, every layout element that
is a patch target or an ancestor of one must have a non-empty authored `id`,
unique recursively within that layout. Projects without localization retain
the current permissive element-ID behavior. Enabling localization fails
validation and extraction with the layout and element location until stable IDs
are persisted; neither path may synthesize IDs from array positions.

```yaml
layouts:
  settingsLayout:
    elements:
      languageTitle:
        content:
          value: 言語
          sourceHash: sha256-...
      playerNameInput:
        placeholder:
          value: 名前を入力
          sourceHash: sha256-...
```

Only text-node `content` and input `placeholder` are patchable in v1. Element
geometry, interactions, resource references, template directives, and IDs are
not patchable. Locale-specific geometry should use locale-specific text-style
overrides or a future explicitly typed layout-variant feature, not arbitrary
patches.

### Control Elements

Control resources use a separate typed family because
`resources.controls[*].elements` are not layout resources.

```yaml
controls:
  quickMenu:
    elements:
      saveLabel:
        content:
          value: セーブ
          sourceHash: sha256-...
      slotNameInput:
        placeholder:
          value: スロット名
          sourceHash: sha256-...
```

The same non-empty-ID, recursive-uniqueness, and compatibility rule applies
independently within each control resource. The same text-node `content` and
input `placeholder` leaves allowed for layout elements are patchable. Keyboard
mappings, actions, payloads, geometry, resource references, template
directives, and IDs remain canonical and cannot be patched.

### Inventory

```yaml
items:
  healingTea:
    name:
      value: 癒やしのお茶
      sourceHash: sha256-...
    description:
      value: 少し元気を回復する。
      sourceHash: sha256-...
inventories:
  player:
    name:
      value: 持ち物
      sourceHash: sha256-...
```

Item tags, limits, use behavior, quantities, and inventory policies are never
patched.

### Image-Gallery Layout Capability

```yaml
layouts:
  cgGalleryLayout:
    imageGallery:
      items:
        sunsetEnding:
          title:
            value: 夕焼け
            sourceHash: sha256-...
          description:
            value: 海辺で迎える最後の夕方。
            sourceHash: sha256-...
          alt:
            value: 夕焼けを背に立つ二人。
            sourceHash: sha256-...
```

Gallery capability data is addressed through its layout resource ID; there is
no separate image-gallery patch domain. Item order, IDs, image references, and
unlocks are not patchable.

### Music Boxes

```yaml
musicBoxes:
  soundtrack:
    name:
      value: 音楽鑑賞
      sourceHash: sha256-...
    tracks:
      mainTheme:
        title:
          value: 新しい道
          sourceHash: sha256-...
        artist:
          value: 作曲者名
          sourceHash: sha256-...

          value: オリジナルサウンドトラック
          sourceHash: sha256-...
        description:
          value: メインテーマ。
          sourceHash: sha256-...
```

Track order, sound IDs, clip bounds, unlocks, and player defaults are not
patchable.

### Scene Replay Catalog

```yaml
sceneReplays:
  memories:
    name:
      value: シーン回想
      sourceHash: sha256-...
    entries:
      firstMeeting:
        title:
          value: 最初の出会い
          sourceHash: sha256-...
        description:
          value: 駅で交わした最初の言葉。
          sourceHash: sha256-...
```

Replay entry pointers, context overrides, restrictions, IDs, and unlocks are
not patchable.

## Resource Overrides

Localized files are typed separately from text patches.

```yaml
resourceOverrides:
  images:
    stationSign:
      fileId: station-sign-ja
      width: 1920
      height: 1080
  videos:
    stationArrival:
      fileId: station-arrival-ja
      fileType: video/webm
      width: 1920
      height: 1080
  spritesheets:
    stationBoard:
      fileId: station-board-ja
  sounds:
    mainTheme:
      fileId: main-theme-ja
  voices:
    chapter1:
      line30Voice:
        fileId: voice-ja-ch1-line30
  fonts:
    dialogueFont:
      fileId: noto-sans-jp
      fileType: font/otf
```

Rules:

- Resource overrides are sparse typed merges over an existing source resource;
  they cannot create a resource ID or change its resource family.
- Images and videos may override `fileId`, `fileType`, `width`, and `height`.
  Spritesheets may override only `fileId`; their canonical `jsonData`, `width`,
  `height`, and `animations` remain source presentation/animation data. A
  localized atlas must preserve the source atlas geometry, frame order, and
  frame count so every canonical animation frame resolves identically. Sounds
  and voices may override only `fileId` and `fileType`, and fonts may override
  `fileId` and `fileType`.
- The source resource must exist and retain its canonical resource ID.
- The source resource plus sparse override must satisfy that resource family's
  full schema.
- Sound behavior fields such as loop/volume/clip timing remain source logic and
  cannot be changed here.
- Localized voices retain scene-local voice IDs.
- Missing override falls through per locale chain to source resource.
- Asset aliases are loaded before locale activation in v1.

## Text Style Overrides

```yaml
textStyleOverrides:
  dialogueText:
    fontId: dialogueFont
    fontSize: 34
    lineHeight: 1.35
    wordWrap: true
    breakWords: true
    align: left
    direction: ltr
```

Overrides are sparse merges into an existing source text-style resource. They
may change font/layout-related style fields and colors, but cannot add a new
text style ID. The merged result is validated through the full text-style
schema before render-state construction.

## Resolution Algorithm

For a localizable leaf:

1. Identify the source leaf through its typed domain path and stable IDs.
2. Build the active chain as the active locale, its declared intermediate
   fallbacks in order, then source locale exactly once. When source is active,
   the chain contains only source.
3. Find the first package in the chain with a valid patch for the exact target.
4. Use its `value`; otherwise use the source leaf.
5. Apply the same chain independently to resource and text-style overrides.
6. Resolve computed variables and template data as normal.
7. Interpolate `${variables.*}`, `${runtime.*}`, and layout loops/conditionals
   after localization.
8. Resolve authored resource IDs into Route Graphics file aliases/styles.

Example:

```yaml
activeLocale: pt-BR
fallbackLocaleIds:
  - pt
```

If a caption exists in `pt` but its image override exists only in `pt-BR`, each
leaf resolves independently: `pt` caption plus `pt-BR` image.

Fallback validation rejects the owning locale, duplicate IDs, unknown locale
IDs, and `sourceLocale` anywhere in `fallbackLocaleIds`. Source need not and
must not be declared because the runtime always appends it once.

## Resolution Architecture

Do not clone and mutate the entire project tree each time a locale changes.

Recommended internal structure:

```text
canonical projectData
        + active package/fallback indexes
        -> localized project resolver
        -> presentation/render selectors
```

The resolver exposes typed operations such as:

```yaml
operations:
  - resolveCharacter(characterId)
  - resolveLine(sectionId, lineId)
  - resolveLayout(layoutId)
  - resolveControl(controlId)
  - resolveItem(itemId)
  - resolveImageGalleryLayout(resourceId)
  - resolveMusicBox(musicBoxId)
  - resolveSceneReplay(replayId)
  - resolveResource(resourceType, resourceId)
  - resolveTextStyle(textStyleId)
```

Resolved values are cloned or immutable snapshots. Caches are keyed by
project-data generation plus locale and are invalidated by `updateProjectData`
or locale change.

Story routing and actions continue reading canonical IDs and logic. Only
approved presentation leaves pass through localization.

## Runtime State and Actions

Active locale is persisted device runtime:

```yaml
global:
  locale: ja-JP
```

Add `locale` to runtime fields, runtime type validation, persisted global
runtime fields, system-state schema, IndexedDB normalization, and layout
template runtime data. Unlike existing static runtime defaults, locale has a
project-dependent initialization default: validated persisted locale, then
`defaultLocale`, then `sourceLocale`.

Public action:

```yaml
setLocale:
  locale: ja-JP
```

Action behavior:

1. validate supported locale and loaded package/assets
2. no-op if already active
3. abort/clear current auto, skip, and next-line-config timers tied to the old
   render generation
4. set and persist device locale
5. invalidate localized resolver caches
6. mark the current text presentation complete so old- and new-language reveal
   fragments cannot mix
7. emit one full render without re-running current line actions
8. after non-aborted render completion, restart only timers valid for the
   unchanged story/modal state

`autoMode` and `skipMode` preferences need not be toggled off; their timer
generations are replaced safely.

### Locale Projection

Layouts receive:

```yaml
localization:
  sourceLocale: en-US
  defaultLocale: en-US
  activeLocale: ja-JP
  direction: ltr
  locales:
    - id: en-US
      name: English
      selected: false
    - id: ja-JP
      name: 日本語
      selected: true
```

The settings layout can loop this array and dispatch `setLocale`. It does not
inspect package contents.

## Template and Computed-Variable Timing

Localization must happen before interpolation:

Source:

```yaml
text: Welcome, ${variables.playerName}.
```

Patch:

```yaml
value: ようこそ、${variables.playerName}。
sourceHash: sha256-...
```

The translated template is selected first, then `playerName` is inserted.

Translation packages may refer only to template paths already allowed for the
source target. Tooling compares placeholder sets and reports missing, added, or
malformed placeholders. A translator cannot introduce `_event`, actions,
condition directives, or arbitrary function calls into plain dialogue text.

Computed variables are evaluated from canonical runtime data, then exposed to
the localized template just like stored variables. Computed definitions are
logic and are not patchable.

## Save, Load, Rollback, and Progress

- Locale is device state, not slot state.
- Saving never serializes translated strings, localized project clones, or
  locale packages.
- Loading a slot retains the currently selected device locale.
- Rollback restores pointer/context state but not locale.
- Viewed line/resource IDs are canonical and language-independent.
- Unlockable IDs are canonical and language-independent.
- Inventory stores item IDs/quantities, not translated names.
- Replay stores canonical section/line IDs.
- Changing locale creates no rollback checkpoint and executes no story action.

This ensures a save made in one locale loads correctly in another.

## Route Graphics Responsibilities

Route Graphics receives only resolved state:

```yaml
type: text
content: 翻訳済みテキスト
textStyle:
  fontFamily: noto-sans-jp
  fontSize: 34
  direction: ltr
```

It owns:

- glyph rendering and font metrics
- wrapping and alignment
- rich-text/furigana layout
- text reveal over resolved graphemes
- input caret/selection behavior
- bidi direction and shaping support

It does not know the active locale or fallback chain.

### RTL Prerequisite

Current Route Graphics text style documents left/center/right alignment but no
public direction/bidi contract. A locale marked `rtl` must not be advertised as
fully supported until Route Graphics adds and browser-validates:

- `textStyle.direction: "ltr" | "rtl" | "auto"`
- correct bidi shaping/order for static and revealing rich text
- RTL-aware wrapping and alignment defaults
- hidden native input `dir` synchronization
- caret movement, selection, placeholder, and IME behavior in RTL input
- mixed-direction text tests

Route Engine passes the locale/style direction; Route Graphics implements it.
Do not reverse strings in Route Engine.

### CJK and Unicode Line Breaking

Locale-specific styles can select fonts and `breakWords`, but correct CJK line
breaking and punctuation rules are renderer/text-layout concerns. Add focused
Route Graphics tests before claiming typography parity. Route Engine's existing
grapheme-based reveal/auto timing should continue to use resolved dialogue text.

## Translator Tooling

The authoritative exchange format remains JSON.

Generate one locale template containing:

```yaml
formatVersion: 1
locale: ja-JP
sourceLocale: en-US
sourceRevision: sha256-...
patches:
  story: {}
resourceOverrides: {}
textStyleOverrides: {}
```

The extractor walks only approved targets and emits stable IDs, source values
for translator context, source hashes, and empty/preserved translation values.
The import/merge tool must preserve existing translations when target ID and
source hash are unchanged.

Reports:

- missing: source target has no translated leaf
- stale: ID exists but source hash changed
- orphan: patch target no longer exists
- invalid: replacement has wrong type or bad reference
- placeholder mismatch
- resource duration/dimension concern
- font/glyph coverage concern where tooling supports it

Source strings are context, not lookup keys. Duplicate source text at different
semantic IDs remains independently translatable.

## Plural, Gender, and Message Formatting

V1 does not introduce ICU MessageFormat or executable plural rules. Authors may
use existing story conditionals/computed variables to select complete source
strings, and each selected string is translated normally.

If structured messages are added later, they must:

- have a JSON AST/data contract
- use locale-aware plural categories
- validate placeholders statically
- run before final text rendering and after variable resolution as explicitly
  designed
- never evaluate arbitrary code

Do not smuggle plural logic into translation strings as JavaScript expressions.

## Schema and Validation

Add structural JSON Schema plus semantic validation for:

- top-level localization metadata
- locale/package uniqueness and source/default existence
- fallback ordering, uniqueness, known IDs, and source/self exclusion
- package/source locale match
- typed patch families and leaf envelopes
- target existence and replacement type
- non-empty, recursively unique layout and control element IDs for localizable
  paths, with legacy compatibility when localization is disabled
- non-empty, line-unique choice IDs when localization is enabled, plus visual
  IDs used as patch targets
- resource override type/reference validation
- merged text-style validation
- placeholder set compatibility
- forbidden logic fields

Unknown package fields fail validation. This is important because a misspelled
target should not silently become a permanently missing translation.

## Error Handling

- Unsupported persisted locale falls back to valid `defaultLocale`, then source.
- Missing package at initialization is a project error for a declared locale.
- Missing individual patch uses the fallback chain.
- Invalid patch leaf fails package validation; it is not partially merged.
- Missing localized asset falls back only if the override itself is absent. If
  an override declares a broken file, asset loading reports an error instead of
  silently hiding a packaging defect.
- Locale change failure leaves active locale and render state unchanged.
- Project update revalidates the current locale; if removed, it atomically
  selects the new default/source locale.

## Implementation Phases

1. Add localization metadata/package schemas and semantic target indexing.
2. Add active locale runtime field, persistence, hydration, and projection.
3. Implement typed text patch resolver and fallback chain.
4. Apply localization to dialogue, choices, forms, visuals, characters,
   layouts, and controls before templating.
5. Add item, image-gallery layout-capability, music, and replay targets.
6. Add resource and text-style overrides with asset validation.
7. Implement safe runtime locale switching and cache invalidation.
8. Build JSON extraction/merge/report tooling.
9. Add Route Graphics direction/bidi support before enabling RTL locales.
10. Add focused system, VT, and browser coverage.

## Test Plan

### Unit and System Coverage

- schema acceptance/rejection for every typed patch family
- missing patch falls to parent/source locale
- fallback order and source/self/duplicate rejection
- same line IDs in different sections resolve independently
- layout and control nested element addressing plus missing/duplicate-ID
  rejection when localization is enabled
- complete dialogue-content array replacement
- empty-string translation versus missing translation
- source hash stale/orphan reports
- placeholder preservation and mismatch rejection/report
- localized character, item, gallery, music, and replay metadata
- localized image/video/spritesheet/audio/voice/font resolution, including
  rejection of spritesheet metadata or geometry overrides
- merged text style validation
- localization occurs before template interpolation
- locale switch does not run line actions or mutate pointer/variables
- active reveal is replaced by a complete new-locale line
- auto/skip timer generations are safe across switch
- save/load/rollback retain active device locale and canonical IDs
- `updateProjectData` cache/locale invalidation

### VT and Browser Coverage

Use isolated fixtures for:

- dialogue locale switch with a variable placeholder
- choice and form placeholder localization
- settings locale list and selected state
- layout and control text plus locale-specific font/style
- image/video/spritesheet file override
- inventory/gallery/music/replay metadata in separate feature pages
- CJK wrap/reveal/auto timing
- RTL static, reveal, mixed-direction, and input behavior after renderer support

Locale switching is a reveal/timing path. It requires both targeted state tests
and an actual browser interaction/re-render path. Do not combine every feature
into one localization fixture.

## Acceptance Criteria

- Canonical project data is never mutated by localization.
- All package formats and actions round-trip through JSON.
- Every patch target is typed and stable-ID based.
- Missing translations always reach source through a validated chain.
- Translated templates preserve and resolve variables after localization.
- Text, assets, fonts, and styles resolve independently and predictably.
- Saves, rollback, inventory, unlocks, and replay retain canonical identity.
- Locale change is render-only with respect to story execution.
- Route Graphics receives resolved presentation state and no locale logic.
- RTL is not claimed until its renderer/input path is implemented and tested.
